## 1. Is the implementation correct?

Yes, against `PLANNER_RUNTIME_PROGRESSION_DESIGN.md`'s own Section 11
"WILL implement" list and, following the Alignment Pass, against
`PLANNER_RUNTIME_CONTRACT_DESIGN.md`'s (Unit D1A) field-level contract
shapes, checked item by item.

`src/contracts/PlanDecision.kt` adds exactly the types Section 11 names,
now shaped exactly as Unit D1A requires: a `PlanCandidate` schema (a real,
`TaskProposal`-field-compatible schema, confirmed unchanged by D1A);
`PlanCandidateId` (confirmed unchanged by D1A); `PlanRejection`, revised by
the Alignment Pass to carry a closed `PlanRejectionReason` enum
(`DUPLICATE_CANDIDATE_ID`/`BLANK_GOAL`/`GOAL_MISMATCH`/`NOT_SELECTED`) plus
a non-blank `detail: String`, replacing the exploratory version's single
free-text `reason: String`; `PlanDecisionResult`
(`Selected`/`NoViableCandidate`, no third variant, confirmed unchanged by
D1A); the `PlanDecision` decision-provider interface, revised by the
Alignment Pass to `suspend fun decide(...)` (AD-010, Model Independence —
so a future model-backed implementation needs no breaking interface
change); `PlanningRequest`, revised by the Alignment Pass to no longer
carry `candidates` (candidates are generated by planning, not supplied by
callers, per D1A); `PlanningSessionResult`
(`Completed`/`Rejected`/`Failed`, confirmed unchanged by D1A, including its
deliberate retention of `Failed.reason` as a free-text `String` — a
one-off terminal summary, not a per-candidate repeatedly-evaluated
signal). A new `PlannerRuntime` interface
(`suspend fun plan(request: PlanningRequest, candidates: List<PlanCandidate>): PlanningSessionResult`)
was added, which `InMemoryPlannerRuntime` now implements — the one net-new
public contract Unit D1A authorised. No `PlannerPolicy` Kotlin type was
added — Section 11 named this as conditional ("if Unit D2's own scoping
determines one is needed"), and this Unit's minimal implementation needs
no configurable bound, so none was introduced; this is a recorded decision
(§5 below), not an oversight.

`src/runtime/DefaultPlanDecision.kt` implements "deterministic candidate
selection" exactly as this unit's own instructions require, and the
Alignment Pass changed nothing about the algorithm itself (per its own
"keep the deterministic Plan Decision algorithm unchanged unless required
by D1A" instruction — D1A required no algorithm change): candidates are
evaluated once, in generation order; the first structurally valid,
non-duplicate candidate wins; every other candidate becomes a
`PlanRejection` with an exact `PlanRejectionReason` value (duplicate id,
blank goal, goal mismatch, or "not selected") plus a non-blank `detail`
string — a strictly stronger, machine-checkable replacement for the
exploratory version's free-text `reason` (see §5 below). No
`RiskEstimate`-based or other weighted ranking exists anywhere in the
class — confirmed by the dedicated `RiskEstimate does not influence
selection` test, which proves a `HIGH`-risk earlier candidate still wins
over a `LOW`-risk later one. Calling `decide` twice with identical
arguments produces an identical result (`selection is deterministic and
repeatable` test) — no clock read, no random value, no external state is
consulted anywhere in the class. `decide` is now `suspend`
(`PlannerRuntime`-interface-driven, per D1A §5); the body itself still
never actually suspends, and no blocking call was introduced.

`src/runtime/InMemoryPlannerRuntime.kt` is now `class InMemoryPlannerRuntime(...) : PlannerRuntime`,
and its `plan(request, candidates)` drives the same progression Section 6
(of the design document) specifies, now taking `candidates` as a second,
explicit parameter rather than reading it off `request.candidates` (the
field the Alignment Pass removed from `PlanningRequest`): identity resolved
before `CREATED`; `CREATED -> CONTEXT_GATHERING -> ANALYSING`, with
`planner.candidate_generated` published once per supplied candidate before
`PlanDecision.decide` is consulted; on `NoViableCandidate`,
`ANALYSING -> FAILED` with `planner.session_failed`; on `Selected`,
`planner.candidate_rejected` for every non-winning candidate (its payload
now carrying the rejection's `reason` enum name and separate `detail`
string, rather than one combined free-text field), then
`ANALYSING -> PROPOSING -> SUBMITTED` around the constructed `TaskProposal`
and its submission via `TaskProposalIntake.submitProposal`
(`InMemoryTaskManagerRuntime`), then `SUBMITTED -> REJECTED` or
`SUBMITTED -> COMPLETED` depending on the returned disposition. This is
confirmed end to end by the `one valid candidate results in a Completed
Planning Session and exactly one Task` test (using the real
`InMemoryTaskManagerRuntime`, proving a real `Task` is actually created,
not merely that a `TaskProposal` value was constructed), by the two
event-sequence tests (`a successful single-candidate Planning Session
publishes the expected planner event sequence`; `a zero-candidate Planning
Session publishes analysis_started then session_failed, never a proposal
event`), and — new in the Alignment Pass — by `InMemoryPlannerRuntime is
usable entirely through the PlannerRuntime interface`, which constructs the
runtime, assigns it to a variable of the interface type, and drives a
complete Planning Session through it, proving the interface is sufficient
on its own and not merely present.

## 2. Are runtime boundaries preserved?

Yes. `InMemoryPlannerRuntime`'s constructor dependencies are exactly what
the design document's Section 3/Section 4 review anticipated:
`IdentityService`, `EventBus`, `TaskProposalIntake` (the already-existing
interface `InMemoryTaskManagerRuntime` implements, unmodified), and an
injected `PlanDecision` (defaulted to `DefaultPlanDecision`) — no new
dependency on the Agent Runtime, Execution Pipeline, Permission Engine,
Tool Registry, or Resource Registry was introduced, and this unit never
calls any of them, directly or indirectly. The Alignment Pass adds exactly
one boundary-relevant change: `InMemoryPlannerRuntime` now implements the
`PlannerRuntime` interface, matching the same architectural pattern
`IdentityService`/`AgentRunCommandChannel`/`TaskProposalIntake` already
establish for every other `InMemory*Runtime` — this is a contract-shape
correction Unit D1A required, not a new dependency or new boundary. AD-005 (Planner Never Creates
Tasks) is unaffected: `InMemoryPlannerRuntime` has no operation that
writes a `Task` record itself — every `Task` still comes into existence
only via `InMemoryTaskManagerRuntime.submitProposal`, exactly as before
this unit existed. AD-002 (Proposal Before Authority) is unaffected: a
`PlanDecision`'s output is a `TaskProposal`, never an `ExecutionRequest`
or a `PermissionDecision` — no path from this unit's code to
`PermissionEngine.evaluate` exists. `TaskProposalIntake`,
`TaskProposalDisposition`, and `TaskManagerRuntimeSpecification.md` are
all unmodified; `InMemoryTaskManagerRuntime`'s own accept-only scope is
unchanged (this unit's `Deferred`/`REJECTED` test paths use
`FakeTaskProposalIntake`, a test-only fixture, precisely so that no change
to `InMemoryTaskManagerRuntime` itself was needed to exercise them).
`PlannerRuntimeSpecification.md` Section 5's ten-state lifecycle is
unmodified; `PlannerSessionLifecycleTransitions`
(`src/contracts/PlannerSessionLifecycle.kt`) models a strict, documented
eight-state subset of it, transcribed rather than invented, and does not
add any edge the specification does not already contain.

## 3. Are tests sufficient?

Sufficient against the explicit list of scenarios this unit's own
instructions required: zero candidates (`DefaultPlanDecisionTest`'s `zero
candidates yields NoViableCandidate...` and
`InMemoryPlannerRuntimeTest`'s `zero candidates results in a Failed
Planning Session...`); one candidate (`one valid candidate is selected...`
and `...results in a Completed Planning Session and exactly one Task`);
multiple candidates (`multiple valid candidates -- the first in generation
order is selected...` and the corresponding end-to-end test); deterministic
selection (`selection is deterministic and repeatable`, `candidate order
determines the winner`, `RiskEstimate does not influence selection`, and
`re-running the same PlanningRequest candidates twice selects the same
winner both times`); invalid candidate rejection (blank goal and
goal-mismatch cases, both at the pure-algorithm level and end to end
through `InMemoryPlannerRuntime`); duplicate candidate handling (including
the three-candidates-sharing-one-id and duplicate-where-the-first-was-
itself-invalid boundary cases); event publication (the full `planner.*`
sequence for both the success and zero-candidate-failure paths, plus a
dedicated test asserting a `planner.candidate_rejected` event's payload);
lifecycle transitions (a direct test of
`PlannerSessionLifecycleTransitions`'s exact modelled edges, including two
explicit negative assertions that no `CREATED -> FAILED` or
`CREATED -> ANALYSING` edge exists, plus `getSessionStatus` reflecting the
terminal status after `plan` returns); and boundary conditions (an
unresolvable initiating Principal, with both a result-shape test and an
own dedicated "publishes no planner events at all" test; resubmission of
the same `planningSessionId` throwing; a `REJECTED` disposition; a
`Deferred` disposition still classifying to `COMPLETED`).

Two new test-only fixtures were added: `FakeTaskProposalIntake`
(`tests/runtime/FakeTaskProposalIntake.kt`, mirroring
`FakePermissionEngine`/`FakeAgentStepSource`'s lambda-based fake precedent)
— needed because `InMemoryTaskManagerRuntime`'s own accept-only scope
cannot produce a `Rejected` disposition for a resolvable owner, nor any
`Deferred`/`Split`/`Merged` disposition at all, and this unit's own scope
explicitly excludes changing `InMemoryTaskManagerRuntime` to make it do
so. No existing test file was modified.

The Alignment Pass added one new test to `InMemoryPlannerRuntimeTest.kt`
(`InMemoryPlannerRuntime is usable entirely through the PlannerRuntime
interface`, §1 above) and updated every existing assertion that previously
substring-matched a rejection's free-text `reason` (e.g.
`reason.contains("duplicate", ignoreCase = true)`) to check the exact
`PlanRejectionReason` enum value instead — a strictly stronger, not
weaker, assertion, per this unit's explicit "update every affected test.
Do not weaken assertions" instruction. Every `@Test` function touching
`DefaultPlanDecision.decide` or `PlannerRuntime.plan` is now wrapped in
`runTest`, since both are `suspend`.

One limitation, disclosed rather than hidden: `PlannerSessionStatus`
values reached only transiently mid-`plan()` (`CREATED`,
`CONTEXT_GATHERING`, `PROPOSING`, `SUBMITTED`) are never directly observed
by a test at that intermediate moment — `plan` is a single, non-suspended-
for-external-input call from a test's point of view, unlike
`InMemoryAgentRuntime`'s multi-command `SUSPEND`/`RESUME` surface, so there
is no natural boundary at which a test could pause and inspect an
in-flight status. This is a property of this unit's own, deliberately
narrower, one-shot design (no `SUSPEND`/`RESUME`/`CANCEL` command channel
for the Planner Runtime — see Section 11's own "will NOT implement" list),
not an untested code path: every transition is still exercised (proven by
the terminal status and event-sequence assertions), just not paused
mid-flight. Android Studio itself has not yet been run against this suite
by a human (see Status above); this review's test-sufficiency finding is a
static read of the test source, not a confirmed green run. The 315/315
figure quoted in Status remains an arithmetic projection, not a verified
result.

## 5. Does implementation comply with approved specifications?

Yes, against the design document's own priority order (Constitution → AD
→ PES-001 → Governance → Sprint Plan → `PlannerRuntimeSpecification.md`),
with implementation choices the design document left open, each recorded
rather than silently decided:

- **No `PlannerPolicy` Kotlin type.** Section 11 named this as
  conditional on Unit D2's own scoping. This implementation needs no
  configurable bound (no `maxPlanCandidates`, no timeout), so none was
  added — unlike `AgentPolicy` in Track C, which Unit C2 did need for
  `maxAgentSteps` enforcement.
- **Candidate generation entirely out of this class's scope.**
  `PlanCandidate`s arrive pre-formed, passed to `PlannerRuntime.plan` as an
  explicit `candidates` parameter (moved off `PlanningRequest` by the
  Alignment Pass, per Unit D1A's finding that bundling `candidates` onto
  `PlanningRequest` contradicted `PlannerRuntimeSpecification.md` Section
  4's own definition of "Planning Request," which never mentions
  candidates); `InMemoryPlannerRuntime` never generates one itself. This is
  not a simplification of the design document — Section 11 and this unit's
  own instructions both explicitly exclude "Planner reasoning" and "LLM
  integration," and the design document's own Section 4 review already
  established that candidate generation is upstream, model-independent
  territory (AD-010).
- **`PlanRejection.reason` as a closed enum, not free text.** The
  exploratory implementation's `reason: String` forced tests to
  substring-match human prose, which is not machine-checkable per this
  unit's own "completely explainable and repeatable" requirement. Unit
  D1A required the fix; the Alignment Pass introduced
  `PlanRejectionReason` (a four-value closed enum) plus a separate,
  non-blank `detail: String` carrying the human-readable specifics — the
  decision logic and its precedence order are unchanged, only how the
  outcome is represented.
- **`PlanDecision.decide` as `suspend`.** The exploratory implementation
  made `decide` a plain (non-suspend) function. Unit D1A flagged this as a
  foreclosed-option risk: a future model-backed or human-in-the-loop
  `PlanDecision` (AD-010, Model Independence; `PlannerRuntimeSpecification.md`
  Section 14) would need to suspend, which would force a breaking
  interface change later if not addressed now. The Alignment Pass made
  `decide` `suspend` without introducing any actual blocking call in
  `DefaultPlanDecision`'s body.
- **Exactly one Task Proposal per Planning Session.** `PlannerRuntimeSpecification.md`
  Section 6 allows "one or more" Task Proposals per Planning Session; this
  unit's `InMemoryPlannerRuntime` only ever constructs one, from the single
  selected `PlanCandidate`. Splitting a Planning Session's output into
  several Task Proposals is explicitly out of Section 11's "will NOT
  implement" list ("Multi-agent planning", "Resource optimisation").
- **An unresolvable initiating Principal is a pre-`CREATED` guard, not a
  `CREATED -> FAILED` transition.** `PlannerRuntimeSpecification.md`
  Section 5's own diagram has no such edge; this was flagged explicitly in
  the design document's own Section 5 review and implemented exactly as
  that review anticipated (`PlannerSessionLifecycleTransitions`'s own
  KDoc restates the reasoning).
- **`Deferred`/`Split`/`Merged` classify to `COMPLETED` without
  implementing their decision logic.** Reading and classifying an
  already-returned `TaskProposalDisposition` value per
  `PlannerRuntimeSpecification.md` Section 5's own rule ("any outcome
  other than every proposal being rejected outright") is not the same as
  implementing the disposition mechanism itself, which remains explicitly
  out of scope (Section 11's "will NOT implement": "`Deferred`/`Split`/`Merge`
  semantics", "Task Proposal disposition handling").

Every "will NOT implement" item named both in the design document's
Section 11 and restated in this unit's own instructions was confirmed
absent: no Memory or World Model reference anywhere in the new code; no
reasoning, heuristic, or model-dependent logic in `DefaultPlanDecision`
(confirmed by the dedicated `RiskEstimate does not influence selection`
test); no `TaskProposalDisposition` handling logic added to
`InMemoryTaskManagerRuntime` (unmodified); no resource selection rule; no
Workflow Engine or Android reference; no new Planner-specific concurrency
mechanism beyond the existing mutex-and-two-phase precedent
(`InMemoryPlannerRuntime`'s own KDoc states this explicitly); no promotion
of `DeterministicPlannerHarness` into `src/` (it remains untouched in
`tests/runtime/`, and `InMemoryPlannerRuntime` is a wholly separate class).

## 8. Remaining Gaps

- `IMPLEMENTATION_GAPS.md` #45 (new, logged alongside this review) — no
  dedicated `planner.session_rejected` event exists for the
  `SUBMITTED -> REJECTED` transition, which is now a real, reachable code
  path (`InMemoryPlannerRuntime`) rather than a purely theoretical one;
  `PlannerRuntimeSpecification.md` Section 11 already records this as an
  Open Question, not decided here.
- `InMemoryTaskManagerRuntime` remains accept-only —
  `Deferred`/`Split`/`Merged`/business-`Rejected` dispositions are still
  unimplemented there (a pre-existing, separately-tracked limitation,
  unchanged by this unit; this unit's own tests reach those classification
  branches only via `FakeTaskProposalIntake`).
- `WAITING_FOR_INPUT` and `CANCELLED` remain unreached by
  `InMemoryPlannerRuntime` — both real, specified states with no
  production code path yet, per `PlannerSessionStatus`'s own documented
  scope.
- No `PlanCandidate` *generation* mechanism (reasoning, heuristic, or
  model-backed) exists anywhere in this repository — `PlanCandidate`s must
  currently be supplied by a caller (a test, today); this remains
  explicitly out of scope for both Unit D1 and Unit D2.
- Android Studio has not yet been run against this suite by a human; the
  315/315 figure in this review's Status section is a static projection,
  not a confirmed result. This session's sandbox had no working
  Kotlin/Gradle toolchain available at any point, including during the
  Alignment Pass itself — this limitation is disclosed here explicitly,
  not worked around, and a human must run the real suite in Android Studio
  and report the actual count before commit.
